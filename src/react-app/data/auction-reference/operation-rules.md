# 경매 브리핑·권리분석 검토 운영규칙

> 목적: 자사 사이트에서 브리핑·권리분석을 자동 생성하고, **담당자(작성)**와 **검수인(검증)**이 같은 건을 서로 다른 관점에서 두 번 봄으로써(크로스체크) 누락·오류를 방어한다.
> 단일 원천: 모든 문구·체크리스트·탐지 규칙은 `경매_특수상황_마스터.yaml` 한 파일에서 나온다. 사이트는 이 파일을 읽어 세 화면(브리핑 / 담당자 체크 / 검수인 체크)을 자동 생성한다.
> 기준일: 2026-06-30 · 마스터 버전: 1.0 · 특수상황 46항목

---

## 1. 전체 구조 한 장 요약

```
[자료 업로드]
     │
     ▼
[게이트1] 자동탐지 ── trigger 매칭 → 특수상황 자동 태깅
     │
     ▼
[게이트2] 담당자 확인·보완 ── 누락 태그 수동 추가 + 담당자 체크리스트 조사·근거 기재
     │
     ▼
[게이트3] 브리핑 생성 ── 브리핑_팩트 + {행위유도} 자동 조립, 본건 사실 보완
     │
     ▼
[게이트4] 검수인 크로스체크 ── 독립 재확인 → 승인 / 반려
     │
     ▼
[발행]  ← 4개 게이트 전부 통과해야만 가능
```

핵심 원칙은 하나다. **같은 위험을 작성자와 검수자가 서로 다른 체크리스트로 두 번 본다.** 담당자 체크는 "조사해라"(현장·관청 확인), 검수인 체크는 "검증해라"(서류 대조·재계산·누락 점검)로 의도적으로 다르게 구성되어 있다.

## 2. 마스터 항목 구조 (필드 7종)

| 필드 | 누가 쓰나 | 누가 보나 | 설명 |
|---|---|---|---|
| `코드` / `명칭` | 시스템 | 전원 | 식별자(예: `LIM-01` 선순위임차인) |
| `자동탐지_트리거` | 시스템 | 시스템 | 업로드 자료에서 이 상황을 자동 태깅하는 키워드·조건 |
| `브리핑_팩트` | 마스터 | 고객 | 고객용 설명(사실) |
| `행위유도` | 마스터 | 고객·담당자 | 검토·확인 등 주의 문구. 브리핑에는 `{ }`로 표시 |
| `담당자_사전조사_체크리스트` | 담당자 | 담당자 | **작성 전** 현장·관청에서 확인할 항목 |
| `검수인_크로스체크_체크리스트` | 검수인 | 검수인 | 담당자 작성물을 **독립적으로 재확인**할 항목 |
| `근거` / `위험도` / `인수여부` | 마스터 | 전원 | 법령·판례, 상/중/하, 낙찰자 부담 요약 |

카테고리 코드: `LIM`(임차·배당) `BLD`(건물·건축) `LND`(토지권리) `AGR`(농지·산지) `REG`(규제·행위제한) `ROD`(도로·통행) `RED`(정비사업) `OWN`(소유·등기특수) `ENC`(유치·담보) `IND`(산업단지) `ETC`(기타).

## 3. 4-게이트 워크플로 (상세)

### 게이트 1 — 자동탐지 (시스템)

업로드된 등기부등본·매각물건명세서·감정평가서 텍스트에서 각 항목의 `자동탐지_트리거`를 매칭해 특수상황을 자동 태깅한다.

예시 규칙:

- 전입세대열람상 전입일 < 말소기준권리일 → `LIM-01 선순위임차인`
- 건축물대장에 "위반건축물" 문자열 → `BLD-01 위반건축물`
- 지목이 전·답·과수원 → `AGR-01 농취증`
- 토지만 매각이고 지상 건물 현존 → `LND-01 법정지상권`
- 토지이용계획확인서에 "개발제한구역" → `REG-01 그린벨트`

**자동탐지는 false negative(놓침)가 가장 위험**하므로 확신 임계치를 낮게 잡아 의심되면 태깅한다. 과태깅은 게이트 2에서 담당자가 제거하면 된다.

### 게이트 2 — 담당자 확인·보완 (담당자)

1. 자동 태깅 결과를 검토하고, 자동탐지가 놓친 특수상황을 **수동으로 추가**한다(여기서 추가 못 하면 시스템은 영원히 모른다).
2. 잘못 태깅된 항목은 사유를 적고 제거한다.
3. 확정된 각 특수상황의 `담당자_사전조사_체크리스트`를 현장·관청 조사로 채운다.
4. **모든 체크 항목에 `근거(서류명·페이지·통화일시·담당부서)`를 필수 입력**한다. 근거 없는 체크는 미완료로 간주한다.

### 게이트 3 — 브리핑 생성 (담당자)

확정된 특수상황의 `브리핑_팩트`와 `{행위유도}`가 자동 조립된다. 담당자는 본건 고유 사실(사건번호·소재지·금액·임차인명 등)만 빈칸에 채운다. 마스터 문구 자체는 임의 수정하지 않는다(수정이 필요하면 4절 변경관리 절차로).

### 게이트 4 — 검수인 크로스체크 (검수인)

1. 검수인은 담당자가 채운 체크 답을 **보지 않은 상태에서** 같은 건의 `검수인_크로스체크_체크리스트`를 먼저 독립적으로 수행한다.
2. 그 다음 담당자 기재 내용과 대조한다. 불일치·근거 부족·누락이 있으면 사유를 적어 **반려**한다.
3. 모든 항목 통과 시에만 승인하며, 승인 즉시 발행 가능 상태가 된다.

---

## 4. 크로스체크 방어 3대 장치

1. **근거 강제** — 담당자·검수인 모든 체크 항목에 출처 입력을 필수화한다. 사후 분쟁 시 "무엇을 근거로 판단했는가"가 기록으로 남아 회사를 방어한다.
2. **돈 직결 항목 이중 계산** — 인수보증금·이행강제금·지료·분담금 등 금액이 걸린 항목(`인수여부`가 "인수"/"조건부 인수"인 항목)은 검수인이 담당자 숫자를 보지 않고 **직접 재계산**한다. 두 숫자가 일치해야 통과.
3. **누락 우선 점검** — 모든 검수 건의 첫 점검 항목은 고정이다: *"자동탐지·담당자가 놓친 특수상황은 없는가?"* 시스템·담당자·검수인 3중으로 누락을 거른다.

## 5. 책임 분담

| 구분 | 담당자 | 검수인 |
|---|---|---|
| 역할 | 작성 | 검증 |
| 행위 동사 | 조사·확인(현장/관청) | 대조·재계산·반려 |
| 산출물 | 브리핑 초안 + 사전조사 근거 | 승인/반려 + 검증 근거 |
| 금액 항목 | 1차 산정 | 독립 재계산 |
| 누락 | 수동 태그 추가 | 누락 최종 점검 |
| 마스터 문구 | 사용(수정 금지) | 부정확 발견 시 변경요청 |

같은 사람이 담당자이자 검수인일 수 없다(자기 검수 금지).

## 6. 위험도별 처리 기준

| 위험도 | 대표 항목 | 처리 |
|---|---|---|
| **상** | 선순위임차인, 위반건축물, 유치권, 법정지상권, 신탁, 선순위가등기, 농취증, 비오톱 | 검수인 필수 + 금액 재계산 + 가능하면 2인 검수 |
| **중** | 분묘기지권, 구분지상권, 재개발, 산업단지, 전용·점용류 | 검수인 필수 |
| **하** | 입목, 미불용지, 농지·산지연금, 관정 | 검수인 약식(누락·근거만 확인) |

## 7. 변경 관리 (마스터 수정 절차)

법령 개정·판례 변경 시 `경매_특수상황_마스터.yaml`만 고치면 세 화면에 자동 반영된다.

1. 변경 제안은 검수인 또는 법무 담당이 사유·근거와 함께 등록한다.
2. `메타.버전`을 올리고 `기준일`을 갱신한다.
3. 영향 받는 항목의 `근거`(법령·판례 날짜)를 함께 갱신한다.
4. 과거 발행물은 발행 당시 버전을 보존한다(소급 변경 금지).

권장 정기 점검 주기: 분기 1회 법령·판례 업데이트 검토.

## 8. 사이트 연동 가이드 (요약)

- 마스터 로드: `경매_특수상황_마스터.yaml` → `항목[]` 배열.
- 브리핑 화면: 태깅된 코드의 `브리핑_팩트` + `행위유도`(중괄호 강조)를 순서대로 렌더.
- 담당자 화면: 태깅된 코드의 `담당자_사전조사_체크리스트`를 체크박스 + 근거입력란으로 렌더.
- 검수인 화면: 같은 코드의 `검수인_크로스체크_체크리스트`를 체크박스 + 근거입력란으로 렌더하되, 담당자 답안은 검수 1차 완료 후 노출.
- 게이트 잠금: 각 게이트의 모든 필수 항목(근거 포함)이 채워지기 전에는 다음 단계 버튼을 비활성화.
